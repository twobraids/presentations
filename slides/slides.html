Socorro

Two Components

  • Breakpad - the client side
  • Socorro - the server side

Breakpad

a joint effort of Google and Mozilla

Breakpad

C library Built into Mozilla Products:

  • Gathers forensic information from a dying instance of Firefox.
  • Interacts with the User
  • Then, if given consent, sends crash information to Socorro

Breakpad

does an amazing thing

  • manages to run as Firefox crashes
  • cannot allocate memory
  • cannot use the stack
  • must manage to send an HTTP POST

Breakpad

HTTP Post to Socorro contains:

  • metadata about the program that crashed
    • ProductName & Version
    • Crashtime, TimeSinceLastCrash, TimeSinceStartup
    • Addons, VideoDrivers, etc.
  • any comments the user is willing to share

Collector

the point of contact between
Breakpad and Socorro

Collector

  • Apache mod-wsgi
  • accepts POSTs from Breakpad
  • filters them
  • stores them for later retrieval

Collector

  • Has to run FAST
  • Accepts 3,000,000 to 5,000,000 crashes per day
  • Uses local file system for storage

Collector

3 layer design

  • HTTP input
  • Rule-based filtration
  • Data storage to a file system

File System Storage

  • 99.999% uptime
  • databases go down
  • 3 days of buffer if the rest of the system fails

CrashMover

it's just a conveyor belt

  • moves crashes from one storage scheme to another
  • doesn't change or interpret

CrashMover

  • move crashes from temporary File System storage into permanent HBase storage
  • push crash IDs into RabbitMQ to tell processors what crashes to process

CrashMover

another 3 layer form

  • read from a file system
  • refrain from changing the data
  • save the crash data to two different datastores

CrashMover

HBase

  • permanent storage for raw crashes
  • allows for analysis using MapReduce
  • more than 100TB of storage on 70 machines

CrashMover

RabbitMQ

  • a queuing system for new crashes
  • tells the processors what to do next
  • saves only the ID of a crash
møøse get along well with rabbits

Processor

the Socorro Workhorse

Processor

  • runs stackwalker, part of Breakpad
  • converts raw crash into processed crash
  • categorizes crashes
  • creates signatures

Processor

Breakpad Stackwalker

  • takes binary crash data
  • using extensive symbol database
  • remarries bare addresses & offsets

Processor

3 layers

  • reads RabbitMQ and HBase
  • transform raw binary to processed
  • saves to 3 stores

Processor

  • multithreaded model
  • 10 12 core machines
  • processed data fans out to many locations

Crontabber

batch job scheduler

  • runs aggregate data analysis code
  • manages 40 different periodic programs
  • includes a dependency rule system
