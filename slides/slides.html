How does it work?

Socorro

Two Components

  • Breakpad - the client side
  • Socorro - the server side

Breakpad

a joint effort of Google and Mozilla

Breakpad

C library collects information:

  • corupt stack
  • corupt heap memory
  • illegal instructions
  • illegal memory dereferencing

Breakpad

does an amazing thing

  • manages to run in a dying process
  • cannot allocate memory
  • cannot use the stack
  • must manage to send an HTTP POST

Breakpad

HTTP Post to Socorro containing:

  • metadata about program that crashed
    • ProductName & Version
    • Crashtime, TimeSinceLastCrash, TimeSinceStartup
    • Addons, VideoDrivers, etc.
  • any comments the user is willing to share

Breakpad

HTTP Post to Socorro containing:

  • a binary blob
    • software stack
    • metadata about hardware
    • metadata about operating system
    • list of loaded libraries

...demo...

Collector

  • Apache modwsgi
  • accepts POSTs from Breakpad
  • filters them
  • stores them for later retrieval

Collector

  • Has to run FAST
  • Accepts 3,000,000 to 5,000,000 crashes per day
  • Uses local filesystem for storage

Collector

conceptually 3 layers

  • HTTP input
  • Rule based filtration
  • Data storage to a filesytem

Collector

CrashMover

it's just a conveyer belt

  • moves crashes from one storage scheme to another
  • doesn't change or interprete

CrashMover

another 3 layer form

  • read from a filesytem
  • refrain from changing the data
  • save the crash data to two different datastores

CrashMover

HBase

  • permanent storage for raw crashes
  • allows for analysis using MapReduce
  • nearly 100TB of storage on 70 machines

CrashMover

RabbitMQ

  • a queuing system for new crashes
  • tells the processors what to do next
  • saves only the id of a crash

Processor

the Work Horse of Socorro

  • runs stackwalker, part of Breakpad
  • converts raw crash into processed crash
  • categorizes crashes
  • creates signatures

Processor

Breakpad Stackwalker

  • takes binary crash data
  • using extensive symbol database
  • remarries bare addresses & offsets

Processor

3 layers

  • reads RabbitMQ and HBase
  • transform raw binary to processed
  • saves to 3 stores

Processor

  • multithreaded model
  • 10 12 core machines
  • fans processed data out to many locations

Crontabber

batch job scheduler

  • runs aggregate data analysis code
  • manages 25 different periodic programs
  • includes a dependency rule system

Crontabber

not 3 layers

  • still uses the same storage modules

Middleware

low level data access

  • RESTful API
  • mostly read-only data from all datastores
  • used by Django for the Socorro UI

Middleware

3 layers

  • HTTP read
  • minor transformations
  • many storage schemes
